# FitMatch Decisions

작성일: 2026-07-07

이 문서는 FitMatch에서 확정된 기획 및 개발 결정을 기록한다. 이후 ChatGPT와 Codex가 프로젝트 맥락을 잃지 않도록, 중요한 방향 변경은 이 문서에 누적한다.

## 2026-07-07

### 결정

하단 탭은 `홈 / 비교 / 기록 / 추천 / 마이`로 유지한다.

### 이유

앱의 핵심 흐름이 대시보드, 상품 비교, 비교 기록, 추천, 사용자 영역으로 나뉘기 때문이다. 내 옷장은 중요하지만 매일 독립 탭으로 접근하기보다 홈과 마이에서 들어가는 관리 기능에 가깝다.

### 영향

하단 탭은 5개로 고정한다. 내 옷장을 독립 탭으로 되돌리지 않는다.

## 2026-07-07

### 결정

홈은 상세 기능 화면이 아니라 대시보드 + 빠른 비교 진입점으로 유지한다.

### 이유

FitMatch의 핵심 행동은 새 상품 URL을 넣고 빠르게 사이즈를 계산하는 것이다. 홈은 앱의 대표 화면이면서 비교 탭으로 빠르게 이동하는 입구여야 한다.

### 영향

홈에는 대표 옷, 최근 비교, 추천 미리보기, 새 상품 비교 진입 카드가 배치된다. 상세 입력 기능은 홈에 넣지 않는다.

## 2026-07-07

### 결정

비교 탭은 URL 입력 중심으로 유지한다.

### 이유

FitMatch의 서비스 가치는 사용자가 쇼핑몰 URL을 넣으면 상품 정보를 자동으로 분석해 추천하는 데 있다. 비교 탭에서 상품명과 사이즈표를 수동 입력하게 하면 핵심 UX가 흐려진다.

### 영향

비교 화면에는 상품 URL 입력과 `사이즈 계산` 버튼을 중심으로 배치한다. 결과는 현재처럼 추천 결과 시트로 표시한다.

## 2026-07-07

### 결정

직접 입력은 비교 탭이 아니라 내 옷장 등록에서 처리한다.

### 이유

직접 입력 데이터는 사용자의 기준 옷 또는 보유 옷 데이터로 관리되어야 한다. 쇼핑 상품 비교 화면에서 직접 입력을 허용하면 기준 옷 데이터와 쇼핑 상품 데이터의 책임이 섞인다.

### 영향

비교 탭의 쇼핑 상품 직접입력 UI와 사이즈표 수동 입력 UI는 제거했다. 사용자가 직접 실측을 입력해야 하는 경우 내 옷장 등록 화면을 사용한다.

## 2026-07-07

### 결정

내 옷장은 독립 탭이 아니라 홈/마이에서 접근한다.

### 이유

내 옷장은 핵심 데이터이지만 앱의 주 사용 행동은 상품 비교다. 하단 탭에는 반복 사용 빈도가 높은 비교, 기록, 추천을 남기고, 옷장 관리는 홈과 마이에서 접근하게 한다.

### 영향

Home의 `내 옷장` 카드와 My의 `내옷장관리` 메뉴에서 `MyClosetView`로 이동한다.

## 2026-07-07

### 결정

무신사를 v0.1 최우선 지원 대상으로 한다.

### 이유

현재 코드에는 무신사 URL 라우팅, OneLink 리다이렉트 처리, 상품 메타데이터 API, actual-size API 파서가 구현되어 있다. 한 쇼핑몰의 전체 흐름을 완성하는 것이 여러 쇼핑몰을 부분 구현하는 것보다 제품 완성도에 유리하다.

### 영향

v0.1에서는 무신사 상품 URL 안정화가 P0이다. 다른 쇼핑몰은 무신사 구조가 안정화된 뒤 추가한다.

## 2026-07-07

### 결정

브랜드와 플랫폼은 구분한다.

### 이유

무신사, 29CM, W컨셉, 쿠팡은 쇼핑 플랫폼이고, 유니클로, 자라, COS, 나이키 같은 곳은 브랜드 또는 공식몰이다. 무신사에 입점한 유니온스튜디오 상품과 유니클로 공식몰 상품은 서로 다른 출처 구조를 가진다.

### 영향

데이터 모델과 UI에서 `sourceType`, `sourceName`, `brandName`을 분리한다.

## 2026-07-07

### 결정

`sourceType`, `sourceName`, `brandName`을 분리한다.

### 이유

상품 출처와 실제 브랜드를 분리해야 추천 기준 옷 선택에서 플랫폼/브랜드/카테고리 우선순위를 정확히 적용할 수 있다.

### 영향

예시는 다음과 같다.

- 무신사 상품: `sourceType = marketplace`, `sourceName = 무신사`, `brandName = 유니온스튜디오`
- 유니클로 공식몰 상품: `sourceType = officialStore`, `sourceName = 유니클로 공식몰`, `brandName = 유니클로`
- 직접 입력 상품: `sourceType = manual`, `sourceName = 직접 입력`, `brandName = 사용자가 입력한 브랜드`

## 2026-07-07

### 결정

추천은 내 옷장 기준이다.

### 이유

FitMatch는 일반적인 사이즈표 계산기가 아니라 사용자가 이미 입어본 옷과 비교하는 서비스다. 추천 품질은 사용자 옷장 데이터의 품질에 의존한다.

### 영향

추천 기준 선택은 `UserFit` 데이터를 중심으로 수행한다. 대표 옷, 같은 세부 카테고리, 같은 브랜드, 같은 플랫폼 여부를 추천 우선순위에 반영한다.

## 2026-07-07

### 결정

추천 결과에는 추천 사이즈뿐 아니라 추천 신뢰도와 이유를 보여준다.

### 이유

사용자는 단순히 “XL 추천”보다 왜 XL인지, 어떤 옷과 비교했는지, 얼마나 믿을 수 있는 추천인지 알아야 한다.

### 영향

`RecommendationHistory`는 추천 점수, 비교 방식, fallback reason, 기준 옷, 상품 실측 차이를 저장한다. 결과 화면은 추천 사이즈, 추천도, 비교 기준, 상품 실측 차이를 보여준다.

## 2026-07-07

### 결정

쇼핑몰 확장은 한 번에 전부 하지 않고 `무신사 → 유니클로 → 29CM` 순서로 한다.

### 이유

쇼핑몰마다 URL 구조, 상품 메타데이터 구조, 사이즈표 표현 방식이 다르다. 하나씩 안정화해야 파서 구조가 무너지지 않는다.

### 영향

새 parser는 `ProductURLParsing` 프로토콜을 따르는 별도 타입으로 추가한다. `ProductURLParserService`는 URL 라우터 역할을 유지한다.

## 2026-07-16

### 결정

옷장 카테고리와 추천 엔진의 비교 속성을 분리한다.

### 이유

카테고리는 사용자가 옷을 찾고 관리하기 위한 분류이며, 실제 비교 가능 여부는 의류 유형, 소매 유형, 봉제 구조와 측정 정의에 의해 결정된다. 같은 카테고리라도 티셔츠와 니트는 다른 비교 대상일 수 있고, 카테고리가 달라도 비교 속성과 측정 정의가 같으면 비교할 수 있다.

### 영향

`Product`와 `UserFit`에 `garmentType`, `sleeveType`, `constructionType`의 안정적인 raw value를 저장한다. 추천 후보 선택은 카테고리 일치 대신 이 비교 속성과 실측 호환성을 우선한다. 카테고리는 옷장 탐색과 측정 항목 정책에 유지한다.

`fitType`은 사용자의 착용 선호와 상품 실측 차이로부터 달라지는 결과 성격이 강하므로 현재 자동 비교의 구조 속성으로 저장하지 않는다. 신축성과 소재도 쇼핑몰별로 일관된 수치나 정의를 확보하기 전까지 추천 필수 조건으로 사용하지 않는다. 추후 검증된 원본 데이터가 확보되면 보조 가중치로 별도 도입한다.
